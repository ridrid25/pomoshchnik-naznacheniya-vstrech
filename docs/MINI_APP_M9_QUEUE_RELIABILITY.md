# Mini App M9 — надёжное согласование

## Основание

Пилот M8 завершён на 9 настоящих заявках. Две заявки, или 22% пилотного потока, перешли в `SLOT_UNAVAILABLE`: время было свободно при отправке, но стало занято до решения администратора.

## Цель M9

Сделать очередь согласования предсказуемой: администратор до нажатия решения видит актуальное состояние слота и не запускает заведомо невозможное подтверждение.

## Пакет M9.1 — живая проверка времени

- API очереди и карточки заявки повторно проверяет рабочее расписание, активные резервы, подтверждённые встречи и Google Calendar;
- собственный резерв проверяемой заявки исключается из конфликта;
- для ожидающей заявки API возвращает `slotAvailable`;
- `canConfirm=false`, если время уже занято;
- в очереди и подробной карточке показано «Время свободно — можно подтверждать» либо «Время уже занято — подтверждение недоступно»;
- для занятого времени остаётся доступно отклонение заявки;
- серверная проверка при подтверждении сохраняется и защищает от конфликта между просмотром и нажатием.
- если Telegram передал подписанные данные запуска в адресе, но клиентский SDK не загрузился, Mini App использует эти данные для создания сессии; подпись и срок действия по-прежнему проверяет backend.
- production-cookie сессии разрешена в Telegram Web iframe через `SameSite=None; Secure; Partitioned`; локальная и тестовая среда сохраняют `SameSite=Strict`.

## Контрольная точка M9.1

1. Свободная ожидающая заявка показывает положительный статус и кнопки подтверждения и отклонения.
2. Если в Google Calendar или расписании появляется конфликт, очередь после обновления показывает занятость.
3. Для занятого слота кнопка подтверждения отсутствует, а отклонение остаётся доступным.
4. Прямой запрос подтверждения всё равно повторно проверяет занятость на сервере.
5. Production health, Telegram menu button и Mini App остаются рабочими.
6. Запуск во встроенном Telegram Web работает даже при недоступности клиентского Telegram SDK.

## Пакет M9.2 — новое время без повторного заполнения

- у закрытой заявки со статусом «Время недоступно» появляется кнопка «Выбрать другое время»;
- тема, формат, длительность, комментарий и email переносятся в новую заявку автоматически;
- пользователь сразу попадает к выбору свободной даты и времени;
- новая заявка получает отдельный ключ идемпотентности и проходит обычное согласование;
- исходная заявка остаётся в архиве и сохраняет историю конфликта.

## Контрольная точка M9.2

1. Для `SLOT_UNAVAILABLE` API возвращает `canRetry=true`.
2. Кнопка повторного выбора не показывается для заявок с другими статусами.
3. После нажатия открывается шаг выбора времени с заполненными данными исходной заявки.
4. Отправка создаёт новую заявку, не изменяя архивную.

## Пакет M9.3 — контроль ожидания решения

- API сообщает, сколько минут каждая заявка ожидает решения;
- после 15 минут заявка помечается как требующая внимания;
- в сводке очереди показаны количество задержавшихся заявок и самое долгое ожидание;
- возраст виден в карточке очереди и в подробностях заявки;
- планировщик раз в пять минут ищет необработанные заявки старше 15 минут;
- администратор получает одну Telegram-сводку с темами заявок и именами пользователей, даже если задержалось сразу несколько заявок;
- успешная отправка фиксируется бизнес-событием `ADMIN_APPROVAL_REMINDER`, поэтому повторного спама нет.

## Контрольная точка M9.3

1. Новая заявка показывает текущее время ожидания и `isAging=false`.
2. После 15 минут API возвращает `isAging=true`, а сводка учитывает заявку в `aging`.
3. Telegram-напоминание отправляется один раз.
4. Если Telegram временно недоступен, напоминание не считается отправленным и будет повторено следующим запуском.
5. После принятия или отклонения заявка больше не участвует в контроле ожидания.
