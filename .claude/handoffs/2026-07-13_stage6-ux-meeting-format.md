# 3. НЕ СРАБОТАЛО

- Восстановление старого кода из GitHub/локальных архивов: репозиторий изначально был пуст, а поиск по уникальным маркерам и стандартным каталогам не нашёл исходники. Поэтому этапы пересозданы по ТЗ и стеку.
- Первоначальный недельный расчёт делал десятки последовательных Google `freebusy` запросов (неделя → день → слоты) и визуально зависал. Увеличение ожидания не решало архитектурную причину; заменено одним диапазонным запросом и 10-секундным API timeout, smoke закрепляет ровно 1 запрос.
- `Start-Process` в текущем Windows sandbox возвращал `Access denied`; `cmd start /b` без перенаправления завершал Node с `EPIPE`, потому что JSON logger писал в закрытый stdout. Рабочий вариант — фоновый запуск с выводом в `NUL`.
- Nodemailer 7 сразу дал high-severity audit advisory. Оставлять его нельзя; обновление до `9.0.3` сохранило утверждённый стек и дало `0 vulnerabilities`.
- Запрет выбора Email при незаполненном SMTP сломал согласованный bot-flow и Stage 3 UAT. Запрет убран: preference сохраняется, доставка остаётся в retry-очереди и не переключается самопроизвольно.
- Идея создавать саму Meet-ссылку только за час отвергнута: при остановке процесса подтверждённая встреча осталась бы без видеоссылки. Ссылка создаётся вместе с Google event, но пользователю раскрывается в одноразовом напоминании за час.
- Встраивать весь личный календарь в Telegram-карточку не стали: это дублирует Google UI и повышает риск раскрытия личных событий. Вместо этого backend показывает результат повторной проверки слота, а администратор получает кнопку открытия точного дня в своём Google Calendar.
- После добавления сущностей старые smoke-тесты жёстко ожидали 6 шаблонов/4 миграции. Ожидание шаблонов обновлено до 8, число миграций вычисляется динамически.
