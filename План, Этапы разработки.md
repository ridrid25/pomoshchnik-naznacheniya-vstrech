## Этап 3. Пользовательский и админский bot-flow

### Цель

Построить рабочий Telegram-интерфейс для пользователя и администратора.

### Что я реализую

- стартовое меню;
- сценарий записи:
  - выбор длительности;
  - выбор недели;
  - выбор даты;
  - выбор времени;
  - ввод или подтверждение email;
  - ввод названия встречи;
  - ввод комментария;
  - подтверждение отправки;
- кнопки `Назад` и `Отмена`;
- раздел `Мои заявки`;
- пункт `Канал уведомлений` с показом текущего выбора и переключением `Telegram`/`Email`;
- административное меню;
- список заявок;
- действия `подтвердить`, `отклонить`, `заблокировать`, `разблокировать`, `открыть настройки`.

### Что должно быть на выходе

- пользователь проходит сценарий записи пошагово;
- пользователь может вернуться назад;
- администратор видит свои действия только со своего Telegram ID;
- бот различает роли.

### Что я проверяю сам

- прохожу все пользовательские сценарии;
- проверяю все кнопки навигации;
- проверяю сохранение чернового состояния;
- проверяю ограничение админ-доступа по Telegram ID.

### Что нужно от вас

- ваш Telegram ID для роли администратора;
- проверка админ-меню с вашего аккаунта.

### Как вы проверяете

1. Открываете бота как обычный пользователь.
2. Проходите сценарий записи.
3. На нескольких шагах нажимаете `Назад`.
4. Проверяете, что можно менять прошлый выбор.
5. Нажимаете `Отмена` и проверяете, что сценарий сбрасывается.
6. Заходите с админского аккаунта.
7. Проверяете, что админ-меню доступно только вам.

### Какие логи должны быть

- старт сценария записи;
- переход между шагами;
- возврат назад;
- отмена сценария;
- открытие админского меню;
- отказ в админ-доступе;
- ошибки callback-сценариев.

## Этап 8. CI/CD автодеплой из GitHub на VPS

### Цель

Автоматизировать обновление бота на сервере после каждого `push` в `GitHub`.

### Что я реализую

- workflow GitHub Actions для production deploy;
- серверный deploy-скрипт;
- проверки обязательных CI-переменных;
- документацию с пошаговой ручной настройкой.

### Что должно быть на выходе

- каждый `push` в `master` автоматически запускает деплой;
- на VPS подтягивается актуальный код;
- контейнеры пересобираются и поднимаются;
- есть health-check после деплоя;
- процесс воспроизводим по инструкции.

### Что я проверяю сам

- workflow-файл валиден синтаксически;
- скрипт деплоя исполняемый и учитывает отсутствие `.env.production`;
- документация закрывает настройки ключей/секретов/переменных.

### Что нужно от вас

- добавить SSH-ключи/Secrets/Variables в GitHub;
- выполнить первый тестовый `push` в `master`;
- подтвердить успешный run в Actions.

### Как вы проверяете

1. Настраиваете Secrets/Variables по `docs/AUTO_DEPLOY_GITHUB.md`.
2. Делаете commit и push в `master`.
3. Проверяете зеленый статус workflow `Deploy Production`.
4. Проверяете на VPS, что обновился commit и контейнеры `healthy`.

### Какие логи должны быть

- GitHub Actions log шага `Deploy on VPS`;
- `docker compose ps` на сервере;
- `docker compose logs app` на сервере;
- `GET /health` после деплоя.

### Критерий завершения

Автодеплой работает стабильно: push в `master` обновляет production без ручного запуска команд на сервере.

## Отдельный трек: Telegram Mini App

Чтобы не смешивать завершенные этапы production-бота и новый контур работ по Mini App, отдельный план Mini App ведется в файле `План разработки Mini App.md`.

Причины такого разделения:
- текущий файл уже фиксирует закрытые этапы `0-8` по рабочему bot-flow;
- Mini App будет развиваться отдельными итерациями и потребует более частых обновлений статуса;
- отдельный файл снижает риск случайно переписать или исказить ранее согласованный план по боту.
